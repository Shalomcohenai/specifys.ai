1) State Analysis (Core Runtime State)
==================================================

Global state currently owned by spec-viewer-main.js:

- currentSpecData
  - Purpose: canonical spec object (id, title, overview/technical/market/design/architecture/visibility/prompts/diagrams/mockups/mindMap, status map, overviewApproved, metadata).
  - Access pattern:
    - Read/write across nearly all tabs.
    - Updated by loadSpec(), Firestore onSnapshot, polling fallback, and generation/retry flows.
    - Mirrored to window.currentSpecData via updateCurrentSpecData().

- currentTab
  - Purpose: current active tab name.
  - Access pattern:
    - Written in window.showTab().
    - Read by displayOverview/displayTechnical/displayMarket/displayDesign for subsection refresh behavior.

- isLoading
  - Purpose: coarse lock for long-running actions.
  - Access pattern:
    - Used by approveOverview(), generateDiagrams(), generatePrompts(), and retry guards.
    - Cross-cutting concern across prompt/diagram/general generation.

- specUnsubscribe, specPollInterval
  - Purpose: realtime lifecycle for spec updates (Firestore listener + polling backup).
  - Access pattern:
    - Set/cleared in loadSpec(), startSpecStatusPolling(), beforeunload/unload cleanup.
    - Drives tab content/status updates indirectly through displaySpec/updateStatus/updateNotificationDot.

- userHasProAccess
  - Purpose: gate editable/pro features (edit, mockup, visibility-engine).
  - Access pattern:
    - Derived by checkProAccess() using /api/v3/credits + users.plan fallback.
    - Used by edit and tab enable/disable flows.

- diagramsData + fullscreen/pan-zoom state
  - diagramsData, currentZoom, currentPanX, currentPanY, isDragging, dragStartX, dragStartY
  - Purpose: legacy standalone diagrams rendering/repair/fullscreen interaction state.
  - Access pattern:
    - Used by displayDiagramsFromData(), displayDiagrams(), renderSingleDiagram(), refreshDiagram(), repairDiagram(), openFullscreen().

- mind map state
  - drawflowInstance, mindMapGenerated
  - Purpose: mind map lifecycle and Drawflow rendering state.
  - Access pattern:
    - Used by initializeMindMapTab(), generateMindMap(), displayMindMap().

- edit mode state
  - isEditMode, originalHTML
  - Purpose: overview inline edit flow.
  - Access pattern:
    - Used by toggleEditMode(), saveOverviewEdit().

- auxiliary UI state
  - currentProgressMessageIndex, window.fullPromptText, window.integrationsText
  - Purpose: progress/prompt copy UX.

Cross-tab state coupling (high):
- status map inside currentSpecData.status is the main coupling mechanism:
  - technical/market/design/architecture/visibility/prompts/diagrams/mockup/chat states gate tab availability and UI rendering.
- updateNotificationDot(), updateTabLoadingState(), updateExportCheckboxes() are shared side-effect hubs.

Move in:

  - markTabAsViewed()

Keep out (for now):

3) Prompt <-> Diagram Dependency Map
==================================================

Direct function-level calls (strict):
- Prompt logic calling Diagram logic: none found.
  - generatePrompts()/generateSingleStage()/displayPrompts* do not call displayDiagrams(), renderSingleDiagram(), repairDiagram(), etc.
- Diagram logic calling Prompt logic: none found.
  - generateDiagrams()/displayDiagrams*/repairDiagram* do not call generatePrompts()/displayPrompts*.

Indirect coupling (shared orchestration/state):
- Both depend on:
  - currentSpecData (shared spec/status source)
  - isLoading (global execution lock)
  - updateTabLoadingState()
  - updateNotificationDot()/updateStatus()
  - showNotification()
  - getAuxHeaders()/window.api
- Both persist back into spec document and trigger downstream side effects:
  - firebase.firestore().collection('specs').doc(...).update(...)
  - triggerOpenAIUploadForSpec(specId)
  - updateExportCheckboxes()

Implication:
- PromptEngine and DiagramEngine can be extracted independently if shared hooks/state adapter are introduced first (via DataService + small UI/status adapter).

==================================================
4) Migration Order (Minimize Breaking Changes)
==================================================

Recommended extraction order:

1. DataService.js (first)
   Why first:
   - Highest fan-in for side effects (Firestore, API, polling, listener cleanup).
   - Enables all later modules to call a stable adapter instead of touching firebase/window.api directly.
   - Minimizes regression risk by centralizing lifecycle and state mutation first.

2. TabManager.js
   Why second:
   - Behavior is mostly DOM routing; low backend coupling.
   - Easy compatibility bridge: keep window.showTab pointing to TabManager.showTab.
   - Reduces global state writes (currentTab) before moving feature engines.

3. DiagramEngine.js
   Why third:
   - Large but relatively self-contained domain (diagrams + mermaid + mindmap).
   - Already structured around explicit function cluster and internal state (diagramsData + fullscreen state).
   - Benefits early from DataService abstraction for save/update/repair endpoints.

4. PromptEngine.js (last among requested modules)
   Why last:
   - Touches widest orchestration surface (staged flow, retries, progress UI, status transitions, section generation fallbacks).
   - Depends on stable status/tab/data contracts; extracting after DataService + TabManager + DiagramEngine reduces coupling churn.

Compatibility strategy throughout:
- Keep window.* function names stable in coordinator/bridge while delegating internals to new modules.
- Migrate by facade first (forwarding calls), then move internals.
- Preserve current DOM IDs and inline onclick contracts until all extractions are complete.

==================================================
5) Risk Notes (for implementation phase)
==================================================

- currentSpecData writes are scattered and not transactional; introduce a single state adapter (getState/setState/patchStatus) before deep extraction.
- isLoading is global; split into scoped locks (promptLoading, diagramLoading, sectionGenerationLoading) to avoid cross-feature blocking.
- Multiple modules currently call notification/status helpers directly; define a shared UI adapter contract to prevent circular dependencies.
- Firestore listener currently triggers both state mutation and rendering; split to:
  - DataService emits events
  - feature modules render on event payloads.
